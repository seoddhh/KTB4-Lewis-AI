# HTTP에 대한 정리

사이트에 들어가기전 항상 붙이는 HTTP에 대해 개발자라면 적어도 이게 어떤 의미인지, 왜 만들어졌는지는 알아야 한다.

우선 HTTP를 위키피디아에 검색하면 ```HTTP는 클라이언트와 서버 사이에 이루어지는 상호 대화를 위한 요청-응답 프로토콜이다. 예를 들면, 클라이언트인 웹 브라우저가 HTTP를 통하여 서버로부터 웹페이지(HTML)나 그림 정보를 요청하면, 서버는 이 요청에 응답하여 필요한 정보를 해당 사용자에게 전달하게 된다. 이 정보가 모니터와 같은 출력 장치를 통해 사용자에게 나타나는 것이다.```라고 써있다.

지금 우리가 사용하는 웹기반의 동작 방식은 ```클라이언트(사용자가 보는 화면)```와 ```서버```의 방식으로 이루어진다 이때 클라이언트에서 보내는```요청(Request)```에 따라 서버는 ```응답(Response)```을 한다. 정리하면 HTTP는 ```"인터넷 상의 애플리케이션끼리는 이런 형식으로 요청하고 이런 형식으로 응답합시다"``` 라는 공통된 약속이다

그럼 자연스럽게 사용자는 자신이 어떤 요청을 하는건지 궁금한데, 다음과 같이 정리할수 있다.

```GET```: 서버로부터 특정 리소스의 조회를 요청한다. 서버의 상태를 변경하지 않는 안전한(Safe) 메서드이며, 필요한 파라미터는 URL의 쿼리 스트링(Query String)을 통해 전달한다. 구조상 HTTP 요청 바디(Request Body)를 포함하지 않는 것이 원칙이다.

```POST```: 서버에 새로운 리소스를 생성하기 위해 데이터를 제출한다. 주로 리소스의 상태를 변경하거나 부수 효과(Side Effect)를 일으킬 때 사용하며, 전송할 데이터는 HTTP 요청 바디에 포함하여 전달한다. 멱등성(Idempotency)이 보장되지 않으므로 동일한 요청을 여러 번 보내면 매번 새로운 리소스가 생성될 수 있다.

```PUT```: 목적 리소스를 요청 본문에 담긴 데이터로 전체 교체(Replace)한다. 요청 경로에 리소스가 존재하지 않으면 새로 생성하고, 존재하면 기존 리소스를 완전히 덮어쓴다. 따라서 리소스의 모든 필드를 누락 없이 전송해야 하며, 여러 번 수행해도 결과가 같은 멱등성을 가진다.

```PATCH```: 리소스를 부분적으로 수정(Modify)한다. PUT과 달리 변경하려는 특정 필드의 데이터만 요청 바디에 담아 전송하므로 자원의 효율적인 업데이트가 가능하다. 구현 방식에 따라 멱등성을 만족할 수도 있고, 만족하지 않을 수도 있다.

```DELETE```: 서버의 특정 리소스를 삭제한다. 삭제할 대상을 명확히 식별할 수 있도록 URI 경로에 식별자(ID)를 포함하여 요청하며, 성공적으로 처리되면 리소스가 완전히 제거된다. 여러 번 호출해도 대상이 이미 삭제된 상태가 유지되므로 멱등성을 가진다.

이런 요청을 위한 방법을 ```매서드(method)```라고 한다. 이외에도 여러가지가 있지만 이 5개가 제일 중요하다 

이렇게 클라이언트가 매서드를 서버에 던지면, 서버는 요청을 처리한 뒤에 자기가 일을 잘 처리했는지 문제가 생겼는지 숫자로 된 세 자리 약속 코드로 알려주는데 이것을 ```상태 코드(Status Code)```라 하고 아래와 같은 숫자로 표현된다

 - 1xx (정보): 요청을 받았으며 프로세스 진행 중

 - 2xx (성공): 요청이 정상적으로 처리됨 (예: 200 OK)

 - 3xx (리다이렉션): 완료를 위해 추가 작업 필요 (예: 301 Moved Permanently)

 - 4xx (클라이언트 에러): 잘못된 문법 등 클라이언트 측 오류 (예: 404 Not Found)

 - 5xx (서버 에러): 서버가 유효한 요청을 처리하지 못함 (예: 500 Internal Server Error)


이때 우리가 주고받는 요청과 응답 형식을 조금 더 뜯어보면 ```헤더(Header)```와 안의 내용물인 ```바디(Body)```라는 영역으로 명확하게 나뉘어 있는 것을 알 수 있다. 

```헤더```는 요청이나 응답에 대한 메타데이터(부가 정보)를 담는 공간이다. 여기에는 데이터의 크기(Content-Length), 종류(Content-Type), 인증 토큰, 
그리고 클라이언트가 사용하는 브라우저 정보 등이 포함되며, ```줄바꿈```을 기준으로 바디와 구분된다.

```바디```는 실제로 클라이언트와 서버가 주고받고자 하는 핵심 데이터(Payload)가 들어가는 공간이다. HTML 소스코드, 이미지 파일, 혹은 API 통신에서 주로 사용하는 
JSON 객체 등이 이 바디를 통해 전달되며, 요청 메서드(예: GET)나 상태 코드에 따라 바디가 비어있을 수도 있다. 

자 그렇다면 ```REST API```는 또 뭔지 이게 HTTP에서 왜 나오는건지 궁금한데 REST는 Representational State Transfer의 약자로, HTTP를 잘 활용하기 위한 설계 원칙(아키텍처 스타일)이다. 
쉽게 말하면 "HTTP 메서드와 URL을 이런 방식으로 설계합시다" 라는 약속이다.REST를 따르는 API를 RESTful API라고 부르며, 핵심은 URL이 리소스(명사)를 나타내고, 행위는 HTTP 메서드(동사)로 표현한다는 것이다. 예를 들면 다음과 같다.

GET    /users        → 전체 유저 목록 조회

GET    /users/1      → 1번 유저 조회

POST   /users        → 새 유저 생성

PUT    /users/1      → 1번 유저 전체 수정

PATCH  /users/1      → 1번 유저 일부 수정

DELETE /users/1      → 1번 유저 삭제

URL에 동사를 넣어 /getUser, /deleteUser/1 처럼 설계하는 방식은 RESTful하지 않다. 행위는 메서드가 이미 담당하고 있기 때문이다.
REST API가 널리 쓰이는 이유는 서버와 클라이언트가 명확하게 분리되고, URL만 봐도 어떤 리소스를 다루는지 직관적으로 이해할 수 있기 때문이다. 
다만 REST는 엄격한 표준이 아닌 설계 가이드라인이기 때문에, 팀이나 서비스마다 구현 방식에 차이가 있을수 있다고 한다.

```무상태성(Stateless)과 상태 유지```
HTTP의 중요한 특성 중 하나는 무상태성(Stateless)이다. 서버는 이전 요청에 대한 정보를 기억하지 않는다. 클라이언트가 요청을 보낼 때마다, 서버 입장에서는 그 클라이언트가 처음 접속한 사람인지 알수 없다.
그럼 이런 문제가 생길수 있는데 예를들어 로그인을 했는데 페이지를 이동할 때마다 다시 로그인해야 하는 상황, 내 정보가 초기화 되는 상황, 상품을 장바구니에 담고 다른 페이지를 둘러보다 돌아오면 장바구니가 비어있는 상황 등이다. 
그래서 상태 유지(Stateful)가 필요한 상황이 생기는데 이를 해결하는 방법이 바로 쿠키(Cookie)와 세션(Session)그리고 요즘 많이 쓰이는 토큰방식(JWT)이다.

```쿠키```: 서버가 클라이언트의 브라우저에 작은 데이터를 저장해두는 방식이다. 이후 요청마다 브라우저가 자동으로 이 쿠키를 헤더에 담아 보내면서 서버가 해당 클라이언트를 식별할 수 있게 된다.

```세션```: 상태 정보를 클라이언트가 아닌 서버에 저장하는 방식이다. 서버는 클라이언트에게 세션 ID만 쿠키로 넘겨주고, 실제 데이터는 서버 메모리나 DB에서 관리한다. 보안적으로 유리하지만 서버에 부하가 생긴다.

```JWT(JSON Web Token)```: 서버가 로그인 성공 시 사용자 정보를 담은 토큰을 발급하고, 이후 클라이언트는 매 요청 헤더에 이 토큰을 포함시킨다. 서버는 토큰의 유효성만 검증하면 되기 때문에, 
Stateless 원칙을 최대한 지키면서도 인증 상태를 유지할 수 있다. 현대 API에서 가장 널리 쓰이는 방식이다.

정리하면, HTTP 자체는 무상태지만 실제 서비스는 상태가 필요하기 때문에 이를 보완하는 다양한 방법이 발전해왔다.

HTTP는 지금까지 계속 발전해 왔는데 그 변천사를 살펴보면 다음과 같다

```HTTP/1.1(1997)```: 오랫동안 웹의 표준이었던 버전이다. 하나의 연결로 여러 요청을 처리할 수 있게 되었지만(Keep-Alive), 요청을 순서대로 처리해야 하는 구조적 한계가 있었다. 
앞의 요청이 느리면 뒤의 요청도 기다려야 하는HOL(Head-of-Line) Blocking 문제가 있다.

```HTTP/2(2015)```: 성능 개선에 초점을 맞춘 버전이다. 하나의 연결에서 여러 stream을 동시에 처리 하는것이 가장 큰 특징이다. 
또한 자주 요청될 것 같은 리소스를 미리 클라이언트에 보내는 서버 푸시(Server Push), 중복되는 헤더를 압축하는 헤더 압축이 도입되면서 전반적인 속도가 크게 향상됐다.

```HTTP/3 (2022)```: 기존 HTTP가 TCP 기반이었다면, HTTP/3는 QUIC이라는 새로운 프로토콜 위에서 동작한다. TCP는 연결 수립 과정에서 시간이 걸리고(3-way handshake), 패킷 손실 시 전체가 멈추는 문제가 있었다. 
QUIC은 UDP 기반으로 이런 한계를 극복하고, 연결 수립 시간도 단축시켰다. 아직 전환이 진행 중이지만, 구글이나 유튜브 같은 대형 서비스들은 이미 HTTP/3를 적극적으로 활용하고 있다.
